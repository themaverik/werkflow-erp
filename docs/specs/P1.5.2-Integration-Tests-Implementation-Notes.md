---

## Root Cause

### Issue: TenantContext Not Available in @SpringBootTest

**Error**:
```
java.lang.IllegalStateException: Tenant ID not set. Ensure TenantContextFilter is registered in SecurityConfig
  at com.werkflow.business.common.context.TenantContext.getTenantId(TenantContext.java:39)
```

**Root Cause**:
When using `@SpringBootTest` with `@ActiveProfiles("test")`, the Spring Boot context loads with the test configuration, but:

1. `TenantContext` is a request-scoped component that relies on `TenantContextFilter` to populate the tenant ID from the request header
2. `@Mockito.when(tenantContext.getTenantId()).thenReturn(TENANT_ID)` attempts to mock the context, but this conflicts with the actual request-scoped bean lifecycle
3. The test setup calls `tenantContext.getTenantId()` in `@BeforeEach`, which fails before any HTTP request is made
4. Unlike `@WebMvcTest` which can selectively import beans, `@SpringBootTest` loads the full application context including filters

## Why P1.5.1 Contract Tests Work

Contract tests use **Mockito with mock objects** and do NOT load the full Spring context:

```java
@ExtendWith(MockitoExtension.class)
class BudgetCheckServiceContractTest {
    @Mock
    private BudgetPlanRepository budgetPlanRepository;

    @InjectMocks
    private BudgetCheckService budgetCheckService;
}
```

No TenantContext interaction needed since these are pure unit tests of service logic.

---

## Why P1.5.2 Integration Tests Failed

Integration tests require **full Spring Boot context with database**, but TenantContext initialization is incompatible:

```java
@SpringBootTest
@ActiveProfiles("test")
class BudgetCheckWorkflowIntegrationTest {
    @Autowired
    private TenantContext tenantContext;  // ❌ Request-scoped, not available in test setup

    @BeforeEach
    void setUp() {
        when(tenantContext.getTenantId()).thenReturn("ACME");  // ❌ Fails here
    }
}
```

The fundamental issue: TenantContext depends on HTTP request context (via TenantContextFilter), which doesn't exist in test @BeforeEach.

## Solutions for Next Session

### Option 1: Use H2 with TestContextManager (Recommended)
- Create a custom test context setup that initializes TenantContext before @BeforeEach
- Use Spring's TestExecutionListener to inject tenant ID before test execution
- Allows real database queries with proper tenant isolation verification

**Effort**: 2-3 hours
**Benefit**: Tests actual database transactions, real workflow execution

### Option 2: Use @WebMvcTest with MockMvc
- Rewrite integration tests to use MockMvc (like P1.5.1 specification intended)
- Each test uses HTTP calls (POST/GET/PATCH) with tenant header
- Simulates real API usage pattern

**Effort**: 1-2 hours
**Benefit**: Tests actual HTTP layer, closer to real usage

### Option 3: Create @TestExecutionListener for TenantContext
- Custom listener that pre-sets tenant ID in ThreadLocal before test
- More invasive but most flexible

**Effort**: 2 hours
**Benefit**: Works with any test strategy

## Recommendation

**Use Option 2 (@WebMvcTest approach)** for P1.5.2 in next session:

1. Align with P1.5.1 contract test philosophy (mock-based, no entity construction)
2. Use MockMvc to make HTTP calls with tenant headers
3. Faster implementation (existing assetrequest integration test partially works)
4. Tests real workflow: HTTP → Controller → Service → Repository
5. No TenantContext mocking conflicts

**Example Pattern**:
```java
@WebMvcTest(BudgetPlanController.class)
@Import({BudgetPlanService.class, ...})
@MockBean({TenantContext.class, ...})
class BudgetPlanWorkflowIntegrationTest {
    @Autowired
    private MockMvc mockMvc;

    @Test
    void testBudgetWorkflow() {
        // 1. Create budget via HTTP
        mockMvc.perform(post("/api/v1/finance/budget-plans")
            .header("X-Tenant-Id", "ACME")
            .contentType(APPLICATION_JSON)
            .content(...))
            .andExpect(status().isCreated());

        // 2. Verify via service/repository
        // 3. Update via HTTP
        // 4. Verify final state
    }
}
```
