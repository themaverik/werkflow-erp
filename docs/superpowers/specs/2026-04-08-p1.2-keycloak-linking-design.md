# P1.2: Keycloak Linking — Design Spec

**Date**: 2026-04-08
**Phase**: P1 — Quality & Integration
**Status**: Design Approved
**Author**: Architecture Review

---

## Overview

Enable Admin Service to link Keycloak user IDs to employees after user provisioning in Keycloak. The linking endpoint is idempotent, multi-tenant scoped, and uses standard JWT security.

**New Endpoint**: `PATCH /api/v1/hr/employees/{employeeId}/keycloak-link`

---

## Problem Statement

### Current State
- Employee entity has `keycloakUserId` field (nullable)
- Admin Service provisions users in Keycloak
- No API endpoint to link the provisioned Keycloak user back to the employee record

### Use Case
1. Admin Service creates employee in werkflow-erp: `POST /api/v1/hr/employees` → returns employee with ID
2. Admin Service provisions user in Keycloak with email → returns keycloak-user-uuid
3. Admin Service calls new endpoint: `PATCH /api/v1/hr/employees/{id}/keycloak-link` with keycloak-user-uuid
4. werkflow-erp updates employee record with keycloak user ID
5. Next time employee logs in via Keycloak, werkflow can lookup by keycloakUserId

### Success Criteria
- ✅ Endpoint accepts keycloakUserId in request body
- ✅ Updates employee.keycloakUserId atomically
- ✅ Idempotent (safe to retry)
- ✅ Returns 409 Conflict if employee already linked to different keycloak user
- ✅ Multi-tenant isolated (tenantId from JWT)
- ✅ Secured via Spring Security (standard JWT bearer token)

## API Contract

### Endpoint

```
PATCH /api/v1/hr/employees/{employeeId}/keycloak-link
Authorization: Bearer {jwt-token}
Content-Type: application/json
Idempotency-Key: {uuid} (optional, for deduplication)

Request Body:
{
  "keycloakUserId": "keycloak-uuid-12345"
}

Response: 200 OK
{
  "id": 1,
  "firstName": "John",
  "lastName": "Doe",
  "email": "john.doe@example.com",
  "keycloakUserId": "keycloak-uuid-12345",
  "employmentStatus": "ACTIVE",
  "createdAt": "2026-04-08T10:00:00Z",
  "updatedAt": "2026-04-08T11:00:00Z",
  ... (full EmployeeResponse)
}
```

### HTTP Status Codes

| Code | Condition | Body |
|------|-----------|------|
| `200 OK` | Successfully linked (first time or idempotent retry with same value) | EmployeeResponse with updated keycloakUserId |
| `400 Bad Request` | Validation failed (empty keycloakUserId, invalid format) | ErrorResponse with VALIDATION_FAILED code |
| `404 Not Found` | Employee with given ID doesn't exist | ErrorResponse with DEPARTMENT_NOT_FOUND code |
| `409 Conflict` | Employee already linked to different keycloakUserId | ErrorResponse with CONFLICT code |
| `401 Unauthorized` | No JWT token or token invalid | Standard Spring Security error |

## Idempotency Semantics

**Idempotent**: Calling the endpoint multiple times with **same keycloakUserId** produces same result.

### Scenarios

**Scenario 1: First call (no keycloakUserId set)**
```
Request: PATCH /employees/1/keycloak-link
Body: { "keycloakUserId": "uuid-123" }

Response: 200 OK
Employee.keycloakUserId now = "uuid-123"
```

**Scenario 2: Retry with same value (idempotent)**
```
Request: PATCH /employees/1/keycloak-link
Body: { "keycloakUserId": "uuid-123" }
Idempotency-Key: same-as-first-call

Response: 200 OK (cached by IdempotencyFilter)
Employee.keycloakUserId still = "uuid-123"
No database write (or write is idempotent)
```

**Scenario 3: Attempt to link to different keycloak user (conflict)**
```
Request: PATCH /employees/1/keycloak-link
Body: { "keycloakUserId": "uuid-999" }
(Employee already has keycloakUserId = "uuid-123")

Response: 409 Conflict
{
  "code": "CONFLICT",
  "message": "Employee already linked to different Keycloak user",
  "timestamp": "2026-04-08T11:00:00Z"
}
```

## Implementation Design

### New DTO: KeycloakLinkRequest

```java
@Data
@NoArgsConstructor
@AllArgsConstructor
public class KeycloakLinkRequest {
    @NotBlank(message = "Keycloak user ID is required")
    @Size(max = 255, message = "Keycloak user ID cannot exceed 255 characters")
    private String keycloakUserId;
}
```

### Controller Method

```java
@PatchMapping("/{employeeId}/keycloak-link")
@Operation(
    summary = "Link Keycloak user to employee",
    description = "Associate a provisioned Keycloak user with an employee record. Called by Admin Service after user provisioning."
)
public ResponseEntity<EmployeeResponse> linkKeycloakUser(
    @PathVariable Long employeeId,
    @Valid @RequestBody KeycloakLinkRequest request
) {
    return ResponseEntity.ok(employeeService.linkKeycloakUser(employeeId, request.getKeycloakUserId()));
}
```

### Service Method

```java
public EmployeeResponse linkKeycloakUser(Long employeeId, String keycloakUserId) {
    // 1. Fetch employee
    Employee employee = employeeRepository.findById(employeeId)
        .orElseThrow(() -> new EntityNotFoundException("Employee with ID " + employeeId + " not found"));

    // 2. Validate tenant ownership (multi-tenant isolation)
    String currentTenant = TenantContext.getTenantId();
    if (!employee.getTenantId().equals(currentTenant)) {
        throw new EntityNotFoundException("Employee not found"); // Don't leak tenant info
    }

    // 3. Conflict detection: if already linked to different keycloak user
    if (employee.getKeycloakUserId() != null &&
        !employee.getKeycloakUserId().equals(keycloakUserId)) {
        throw new ConflictException("Employee already linked to different Keycloak user. " +
            "Current: " + employee.getKeycloakUserId() + ", attempted: " + keycloakUserId);
    }

    // 4. Set keycloak user ID (no-op if already set to same value)
    employee.setKeycloakUserId(keycloakUserId);

    // 5. Save (atomic update)
    employee.setUpdatedAt(LocalDateTime.now());
    employeeRepository.save(employee);

    // 6. Return updated response
    return EmployeeResponse.fromEntity(employee);
}
```

### Exception Mapping

Exceptions are caught by GlobalExceptionHandler (P1.1):

| Exception | HTTP Status | Error Code |
|-----------|-------------|------------|
| `EntityNotFoundException` | 404 | DEPARTMENT_NOT_FOUND |
| `ConflictException` | 409 | CONFLICT |
| `MethodArgumentNotValidException` | 400 | VALIDATION_FAILED |

## Multi-Tenant Isolation

The `linkKeycloakUser()` method validates tenant ownership:

```java
String currentTenant = TenantContext.getTenantId();
if (!employee.getTenantId().equals(currentTenant)) {
    throw new EntityNotFoundException("Employee not found"); // Don't leak tenant info
}
```

**Why**: Prevents Admin Service for Tenant A from accidentally (or maliciously) linking employees from Tenant B.

**Behavior**: If caller tries to link employee from another tenant:
- Return 404 Not Found (not 403 Forbidden)
- Don't reveal that the employee exists in another tenant

## Testing Strategy

### Unit Tests

**Test 1: Happy path - link new keycloak user**
- Employee has no keycloakUserId
- Call endpoint with valid keycloakUserId
- Verify: 200 OK, employee.keycloakUserId updated
- Verify: database write occurred

**Test 2: Idempotent - same keycloak user twice**
- Employee already linked to keycloakUserId "uuid-123"
- Call endpoint again with same "uuid-123"
- Verify: 200 OK
- Verify: no database write (or idempotent write)

**Test 3: Conflict - attempt to relink to different keycloak user**
- Employee already linked to "uuid-123"
- Call endpoint with different "uuid-999"
- Verify: 409 Conflict
- Verify: message indicates conflict

**Test 4: Not found - employee doesn't exist**
- Call endpoint with invalid employeeId
- Verify: 404 Not Found

**Test 5: Tenant isolation - prevent cross-tenant linking**
- Setup: Employee in Tenant A, caller authenticated as Tenant B
- Call endpoint
- Verify: 404 Not Found (not 403, to avoid leaking tenant info)

**Test 6: Validation - empty keycloakUserId**
- Call endpoint with empty keycloakUserId
- Verify: 400 Bad Request with VALIDATION_FAILED code

### Integration Tests (Optional)

- Full flow: Create employee → Link keycloak user → Verify linked
- Multi-tenant scenario: Two tenants with employees, verify isolation
