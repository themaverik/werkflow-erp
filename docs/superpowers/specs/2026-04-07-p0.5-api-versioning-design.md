# P0.5 — API Versioning (/api/v1) Design

**Date**: 2026-04-07
**Status**: APPROVED
**Scope**: Change application context path from `/api` to `/api/v1` across all endpoints and documentation

---

## Problem Statement

Currently, the werkflow-erp service serves endpoints at `/api/...` (e.g., `GET /api/hr/employees`). This lacks explicit versioning, making it difficult to support multiple API versions in the future and unclear to consumers what version they're using. The roadmap calls for `/api/v1/` as the standard versioning scheme.

---

## Design

### Architecture

Change the Spring Boot application's `server.servlet.context-path` from `/api` to `/api/v1`. This is a configuration-only change that automatically applies the new base path to all endpoints, Swagger documentation, and health checks. No controller code changes are needed; Spring Boot handles routing transparently.

**Before:**
```
GET /api/hr/employees
GET /api/finance/budgets
Swagger docs at: /api/api-docs
Health check at: /api/health
```

**After:**
```
GET /api/v1/hr/employees
GET /api/v1/finance/budgets
Swagger docs at: /api/v1/api-docs
Health check at: /api/v1/health
```

### Components to Modify

#### 1. **application.yml** (1 line changed)

**File:** `services/business/src/main/resources/application.yml`

```yaml
server:
  port: ${SERVER_PORT:8084}
  servlet:
    context-path: /api/v1    # ← Changed from /api
  error:
    include-message: always
    include-binding-errors: always
```

#### 2. **Swagger Configuration** (2 lines updated)

**File:** `services/business/src/main/resources/application.yml`

```yaml
springdoc:
  api-docs:
    path: /api-docs          # ← Kept the same; will be served at /api/v1/api-docs
  swagger-ui:
    path: /swagger-ui.html   # ← Kept the same; accessed at /api/v1/swagger-ui.html
    enabled: true
    operations-sorter: method
    tags-sorter: alpha
    oauth2-redirect-url: http://localhost:8084/api/v1/swagger-ui/oauth2-redirect.html
                         # ↑ Updated to include /v1/ in the path
  show-actuator: true
```

#### 3. **Documentation Files** (hardcoded URLs updated)

**Files affected:**
- `docs/WERKFLOW_INTEGRATION.md` — Update all curl examples and endpoint URLs
- `docs/Independence-Checklist.md` — Update all example curl commands
- `Implementation-Summary.md` — Update example URLs in the quick-start section

**Pattern:**
```
FROM: http://localhost:8084/api/...
TO:   http://localhost:8084/api/v1/...

FROM: curl -X POST http://localhost:8084/api/inventory/asset-requests \
TO:   curl -X POST http://localhost:8084/api/v1/inventory/asset-requests \
```

### Data Flow

```
HTTP Request → DispatcherServlet → Context Path Routing → Controller Method
    ↓                                    ↓
GET /api/v1/hr/employees    Spring sees context-path: /api/v1
                            Strips it, routes to @RequestMapping("/employees")
                                    ↓
                            EmployeeController.getAll()
                                    ↓
                            Response (version is transparent to code)
```

### Impact Analysis

| Area | Impact | Details |
|------|--------|---------|
| **Controller Code** | NONE | All controllers use relative paths; Spring applies context path automatically |
| **Service Code** | NONE | Business logic unchanged |
| **Repositories** | NONE | Data layer unchanged |
| **Tests** | MINIMAL | Unit tests use mocks; integration tests may use hardcoded URLs and need updates |
| **External Systems** | HIGH | werkflow (Engine, Admin, Portal) must update URLs when calling werkflow-erp endpoints |
| **Docker Compose** | NONE | Service URLs are dynamically determined by Spring config |
| **Backwards Compatibility** | BREAKING | Old `/api/` endpoints will return 404; complete cutover, no grace period |

### Error Handling

- **404 responses** for requests to old `/api/` paths: Spring Boot's default 404 handling applies
- **No custom error handling needed**: Spring transparently handles path routing based on the configured context path
- **Swagger behavior**: "Try it out" feature in Swagger UI will use the new `/api/v1/` path automatically

### Testing Strategy

**Unit Tests:**
- No changes needed; all unit tests use Mockito mocks and don't depend on HTTP paths

**Integration Tests:**
- Update any hardcoded base URLs from `http://localhost:8084/api/` to `http://localhost:8084/api/v1/`
- Example: `when(mockClient.call("http://localhost:8084/api/hr/..."))` → `when(mockClient.call("http://localhost:8084/api/v1/hr/..."))`

**Manual Testing / Documentation Examples:**
- All curl examples in docs must use `/api/v1/`
- Swagger UI verification: confirm accessible at `http://localhost:8084/api/v1/swagger-ui.html`
- Health endpoint: confirm `curl http://localhost:8084/api/v1/health` returns 200 OK
- Sample endpoint: confirm `curl http://localhost:8084/api/v1/hr/employees` works with valid auth

### Non-Goals

- **Multi-version support** (v1, v2, v3): This is a single-version cutover. Future major API changes will be handled separately.
- **Controller-level versioning** (`@RequestMapping("/v1")`): Unnecessary complexity at this stage. Context-path handles it cleanly.
- **Deprecation grace period** for old endpoints: Complete cutover. Callers must migrate immediately.
- **Version negotiation** (Accept headers, etc.): No content negotiation by version. Just a single `/api/v1/` baseline.

## Scope & Constraints

**Scope:**
- Change context path from `/api` to `/api/v1`
- Update Swagger/Springdoc configuration for correct OAuth2 redirect URL
- Update all documentation files with new endpoint URLs

**Constraints:**
- Must not break existing functionality (i.e., all endpoints still work, just at new paths)
- Must be reversible if needed (just a config change)
- No database or schema changes
- No breaking API contract changes (only the URL prefix changes)

**Out of Scope:**
- Updating external services (werkflow Engine, Admin, Portal) that call werkflow-erp — they will update their URLs in parallel
- API response format changes
- Endpoint behavior changes
- Pagination implementation (that's P0.6)
