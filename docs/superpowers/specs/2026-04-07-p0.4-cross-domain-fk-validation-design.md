# P0.4 — Cross-Domain FK Validation Design

**Date**: 2026-04-07
**Status**: APPROVED
**Scope**: Validate foreign key references across HR, Finance, Procurement, and Inventory domains

---

## Problem Statement

Cross-domain foreign keys (e.g., PurchaseRequest → Department) are stored as raw Long IDs without validation. When creating or updating records, the service accepts invalid or non-existent foreign keys without error, leading to referential integrity violations and downstream errors in the workflow orchestration layer (werkflow).

**Current behavior:**
```json
POST /api/v1/procurement/purchase-requests
{
  "requestingDeptId": 99999,  // ← Non-existent department
  "requesterUserId": 123,
  "justification": "..."
}
// Response: 201 Created (WRONG — dept doesn't exist)
```

**Expected behavior:**
- Validate `requestingDeptId` exists in HR domain before creating purchase request
- Validate cross-tenant isolation: department must belong to same tenant
- Return 400 Bad Request if validation fails

---

## Design

### Architecture

A centralized **CrossDomainValidator** service validates foreign key references at the service layer during create/update operations. Called before business logic proceeds.

```
Controller
    ↓
Service.create(request)
    ├→ CrossDomainValidator.validate*(fk, tenantId)  ← NEW
    │  └→ Repository.findByIdAndTenantId()
    │      └→ throw EntityNotFoundException if not found
    ├→ Business logic (save, update)
    └→ Return response
```

### Components

#### 1. CrossDomainValidator Service

**File**: `com/werkflow/business/common/validator/CrossDomainValidator.java`

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class CrossDomainValidator {

    private final DepartmentRepository departmentRepository;
    private final BudgetCategoryRepository budgetCategoryRepository;

    /**
     * Validates that a department exists and belongs to the specified tenant.
     *
     * @param departmentId Department ID (from HR domain)
     * @param tenantId Tenant ID from TenantContext
     * @throws EntityNotFoundException if department not found or wrong tenant
     */
    public void validateDepartmentExists(Long departmentId, String tenantId) {
        if (departmentId == null) {
            return; // Optional FK
        }

        departmentRepository
            .findByIdAndTenantId(departmentId, tenantId)
            .orElseThrow(() -> new EntityNotFoundException(
                "Department not found: id=" + departmentId + " for tenant=" + tenantId));
    }

    /**
     * Validates that a budget category exists and belongs to the specified tenant.
     *
     * @param categoryId Budget category ID (from Finance domain)
     * @param tenantId Tenant ID from TenantContext
     * @throws EntityNotFoundException if category not found or wrong tenant
     */
    public void validateBudgetCategoryExists(Long categoryId, String tenantId) {
        if (categoryId == null) {
            return; // Optional FK
        }

        budgetCategoryRepository
            .findByIdAndTenantId(categoryId, tenantId)
            .orElseThrow(() -> new EntityNotFoundException(
                "Budget category not found: id=" + categoryId + " for tenant=" + tenantId));
    }

    // TODO: Add validators as needed during P0.4 and beyond
    // public void validateVendorExists(Long vendorId, String tenantId) { ... }
    // public void validateEmployeeExists(Long employeeId, String tenantId) { ... }
    // public void validateAssetCategoryExists(Long categoryId, String tenantId) { ... }
    // public void validateAssetDefinitionExists(Long defId, String tenantId) { ... }
    // public void validateBudgetPlanExists(Long planId, String tenantId) { ... }
}
```

#### 2. Service Layer Integration

**PurchaseRequestService** (`procurement/service/PurchaseRequestService.java`):
- Inject `CrossDomainValidator`
- In `createPurchaseRequest()`: validate `requestingDeptId` before saving
- In `updatePurchaseRequest()`: validate `requestingDeptId` if updated

**CustodyRecordService** (`inventory/service/CustodyRecordService.java`):
- Inject `CrossDomainValidator`
- In `createCustodyRecord()`: validate `custodianDeptId` before saving
- In `updateCustodyRecord()`: validate `custodianDeptId` if updated

#### 3. Error Handling

Validation failures throw `EntityNotFoundException`, which is already caught by Spring and mapped to:
- **HTTP 404** (current behavior) — may change with global error handler in P1.1
- **Message**: "Department not found: id=99999 for tenant=ACME"

**Note**: P1.1 (API Contract Standardization) will introduce a uniform error response format. This design uses existing patterns.

#### 4. Testing

**Unit tests** (`common/validator/CrossDomainValidatorTest.java`):
- Happy path: valid FK exists → no exception
- Error path: FK not found → throws EntityNotFoundException
- Cross-tenant isolation: FK exists but different tenant → throws EntityNotFoundException
- Optional FK: null departmentId → no validation

**Integration tests** (in service test suites):
- `PurchaseRequestServiceTest.createWithInvalidDepartmentId()` → 400/404 error
- `CustodyRecordServiceTest.createWithInvalidDepartmentId()` → 400/404 error

## Scope & Non-Goals

**In Scope**:
- Validate Department FK (used by PurchaseRequest, CustodyRecord, Expense, etc.)
- Validate BudgetCategory FK (used by BudgetLineItem, Expense)
- Cross-tenant isolation for both validators
- Service-layer integration for PurchaseRequest and CustodyRecord

**Out of Scope** (marked TODO):
- Vendor FK validation
- Employee/User FK validation
- AssetCategory/AssetDefinition FK validation
- BudgetPlan FK validation
- Cascade logic (that's business logic, not validation)
- Global error response format (that's P1.1)

**Why TODO approach**:
- Establishes the pattern: each service injects validator, calls validate*() before create/update
- Easier to add validators incrementally as other features encounter FKs
- Reduces initial implementation scope while maintaining extensibility

---

## Architecture Decision Record

This design follows the **multi-tenant, service-layer validation** pattern established in P0.1:
- Validation is pushed to the service layer (not controller, not DTO-level annotations)
- All validations are tenant-scoped: FK must belong to current tenant
- Error handling uses existing `EntityNotFoundException` pattern
- Extensible: new validators added to CrossDomainValidator without changing service signatures

**See also**: `docs/ADR-001-Service-Boundary-Architecture.md` (update with FK validation pattern)

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Service layer bloated with validators | Keep validators minimal; if >5 methods, create sub-validators by domain |
| N+1 queries from validator lookups | Each validator does 1 query; could cache in future if needed (P1.3) |
| Existing data has invalid FKs | Data is new (pre-MVP); assume all existing FKs are valid |
| Validator not called in all paths | Code review checklist: new create/update methods must call validator |
