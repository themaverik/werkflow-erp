# P1.4: Number Generation & Collision Prevention — Design Spec

**Date**: 2026-04-08
**Phase**: P1 — Quality & Integration
**Status**: Design Approved
**Author**: Architecture Review

---

## Overview

Replace timestamp-based document numbering (`PR-1712345678901`) with a collision-proof sequence format that includes tenant isolation, human-readable timestamps, and monotonic counters.

**New Format**: `{PREFIX}-{TENANT_ID}-{TIMESTAMP_MS}-{SEQ:05d}`

Example: `PR-ACME-1712345678901-00042`

---

## Problem Statement

### Current Implementation (Broken)
- **Format**: `PR-1712345678901` (just prefix + millisecond timestamp)
- **Issues**:
  - Collisions possible if 2+ PRs created in same millisecond
  - Not tenant-aware (all tenants share timestamp namespace)
  - Not sortable by tenant or creation sequence
  - Difficult to debug ("which PR number is this?")

### Success Criteria
- ✅ Zero collisions within a tenant
- ✅ Tenant isolation (ACME's numbering independent from TECH)
- ✅ Human-readable (shows tenant, creation time, sequence)
- ✅ Backward compatible with test data
- ✅ No external dependencies (pure PostgreSQL sequences)

---

## Solution Design

### 1. Number Format

**Pattern**: `{PREFIX}-{TENANT_ID}-{TIMESTAMP_MS}-{SEQ:05d}`

| Component | Length | Source | Example |
|-----------|--------|--------|---------|
| PREFIX | 2-3 | Constant | `PR`, `PO`, `GRN` |
| TENANT_ID | Variable | `TenantContext.getTenantId()` | `ACME`, `TECH` |
| TIMESTAMP_MS | 13 | `System.currentTimeMillis()` at creation | `1712345678901` |
| SEQ | 5 | PostgreSQL sequence `nextval()` | `00042` |

**Full Examples**:
- `PR-ACME-1712345678901-00042` (Purchase Request #42, ACME tenant, created at timestamp)
- `PO-TECH-1712346789012-00001` (Purchase Order #1, TECH tenant)
- `GRN-ACME-1712345678901-00100` (Goods Receipt #100, ACME tenant)

### 2. Sequence Strategy

**One sequence per document type per tenant** (lazy creation):

**Database Sequences**:
```
pr_seq_ACME       — PurchaseRequest counter for ACME (starts at 1)
pr_seq_TECH       — PurchaseRequest counter for TECH (starts at 1)
po_seq_ACME       — PurchaseOrder counter for ACME (starts at 1)
grn_seq_ACME      — Receipt counter for ACME (starts at 1)
... etc
```

**Naming Convention**: `{doctype_abbrev}_seq_{TENANT_ID_UPPERCASE}`

| Document Type | Abbreviation | Example Sequence Names |
|---|---|---|
| PurchaseRequest | `pr` | `pr_seq_ACME`, `pr_seq_TECH` |
| PurchaseOrder | `po` | `po_seq_ACME`, `po_seq_TECH` |
| Receipt | `grn` | `grn_seq_ACME`, `grn_seq_TECH` |

**Note**: Tenant ID is always uppercase in sequence names (e.g., if tenantId="acme", sequence is `pr_seq_ACME`). This avoids PostgreSQL identifier quoting issues.

**Sequence Creation**:
- **When**: Lazy, on first document creation for a tenant
- **How**: SQL `CREATE SEQUENCE IF NOT EXISTS` in application code
- **Concurrency**: PostgreSQL sequences are inherently atomic; no locking needed
- **Lifespan**: Never resets (monotonic across entire tenant history)

### 3. Number Generation Logic

**Java Implementation Pattern**:

```java
private String generatePrNumber(String tenantId) {
    // 1. Ensure sequence exists (lazy creation)
    sequenceService.ensureSequenceExists(tenantId, "pr");

    // 2. Get next sequence value
    long seqValue = sequenceService.nextValue(tenantId, "pr");

    // 3. Capture timestamp at creation time
    long timestamp = System.currentTimeMillis();

    // 4. Format as PR-{tenantId}-{timestamp}-{seq:05d}
    String prNumber = String.format("PR-%s-%d-%05d", tenantId, timestamp, seqValue);

    return prNumber;
}
```

**SequenceService** (new, responsible for sequence management):
- `ensureSequenceExists(tenantId, docType)` — Create if not present
- `nextValue(tenantId, docType)` — Get next sequence value via SQL `nextval()`

### 4. Collision Analysis

**Scenario 1: Multiple PRs created in same millisecond**
```
Thread A: timestamp=1712345678901, seq=1 → PR-ACME-1712345678901-00001
Thread B: timestamp=1712345678901, seq=2 → PR-ACME-1712345678901-00002
```
✅ **No collision** — sequences are unique within the millisecond

**Scenario 2: Two tenants (ACME vs TECH)**
```
ACME: PR-ACME-1712345678901-00042
TECH: PR-TECH-1712345678901-00042
```
✅ **No collision** — tenant ID prefix distinguishes them

**Scenario 3: Next millisecond**
```
Old: PR-ACME-1712345678901-00042
New: PR-ACME-1712345678902-00001
```
✅ **No collision** — different timestamp component

**Collision probability**: Theoretical max is 2+ PRs per millisecond for same tenant. In practice, highly unlikely (would require ~1000 concurrent requests).

## Database Migration (Flyway V23)

### Migration Strategy

1. **Create sequence base infrastructure**
   - Helper function to create tenant-specific sequences
   - Base sequences for pre-existing tenants (if any)

2. **Backfill existing numbers** (test data)
   - Query all existing `prNumber`, `poNumber`, `receiptNumber` with current format
   - Extract tenant ID from JWT/header OR default to "DEFAULT"
   - Reassign new format with migration timestamp (now)
   - Update in place

3. **Document for production**
   - No data loss (just reformatting)
   - Safe to run on empty database (no backfill needed)

### Migration SQL (Pseudocode)

```sql
-- V23__Add_sequence_based_numbering.sql

-- 1. Create helper function (used by application, not directly in migration)
CREATE OR REPLACE FUNCTION create_tenant_sequence(p_tenant_id VARCHAR, p_doc_type VARCHAR)
RETURNS VOID AS $$
DECLARE
  v_seq_name VARCHAR;
  v_tenant_upper VARCHAR;
BEGIN
  v_tenant_upper := UPPER(p_tenant_id);
  v_seq_name := p_doc_type || '_seq_' || v_tenant_upper;  -- e.g., pr_seq_ACME
  EXECUTE format('CREATE SEQUENCE IF NOT EXISTS %I START WITH 1', v_seq_name);
END;
$$ LANGUAGE plpgsql;

-- 2. Backfill existing PurchaseRequests (test data preservation)
-- Format: PR-{tenant_id}-{timestamp_ms}-{seq_05d}
-- Migration timestamp used for all old records
WITH numbered_prs AS (
  SELECT id, tenant_id,
         ROW_NUMBER() OVER (PARTITION BY tenant_id ORDER BY id) AS seq_num
  FROM purchase_requests
  WHERE pr_number NOT LIKE 'PR-%-%-%--%'  -- Skip if already new format
)
UPDATE purchase_requests pr
SET pr_number = 'PR-' || UPPER(np.tenant_id) || '-' || (EXTRACT(EPOCH FROM NOW()) * 1000)::BIGINT || '-' || LPAD(np.seq_num::TEXT, 5, '0')
FROM numbered_prs np
WHERE pr.id = np.id;

-- 3. Similar backfill for PurchaseOrders
WITH numbered_pos AS (
  SELECT id, tenant_id,
         ROW_NUMBER() OVER (PARTITION BY tenant_id ORDER BY id) AS seq_num
  FROM purchase_orders
  WHERE po_number NOT LIKE 'PO-%-%-%--%'
)
UPDATE purchase_orders po
SET po_number = 'PO-' || UPPER(np.tenant_id) || '-' || (EXTRACT(EPOCH FROM NOW()) * 1000)::BIGINT || '-' || LPAD(np.seq_num::TEXT, 5, '0')
FROM numbered_pos np
WHERE po.id = np.id;

-- 4. Similar backfill for Receipts
WITH numbered_receipts AS (
  SELECT id, tenant_id,
         ROW_NUMBER() OVER (PARTITION BY tenant_id ORDER BY id) AS seq_num
  FROM receipts
  WHERE receipt_number NOT LIKE 'GRN-%-%-%--%'
)
UPDATE receipts r
SET receipt_number = 'GRN-' || UPPER(np.tenant_id) || '-' || (EXTRACT(EPOCH FROM NOW()) * 1000)::BIGINT || '-' || LPAD(np.seq_num::TEXT, 5, '0')
FROM numbered_receipts np
WHERE r.id = np.id;
```

**Migration Notes**:
- Lines 2-12: Helper function created for application use (Java code calls this)
- Lines 15-24: Backfill PurchaseRequests with new format
- Repeats for PurchaseOrders (lines 26-33) and Receipts (lines 35-42)
- All use migration timestamp (now) for consistency
- Tenant ID normalized to uppercase in sequence names
- Sequence values (00001, 00002, etc.) assigned based on row order

## Error Handling

### Sequence Creation Failures
**Scenario**: `CREATE SEQUENCE` fails (e.g., permission denied)
**Handling**:
- Catch `SQLException` in `ensureSequenceExists()`
- Log error, throw `SequenceCreationException`
- Bubble to controller as 500 error with message "Unable to generate document number"

### Concurrent Sequence Access
**Scenario**: Multiple threads call `nextValue()` simultaneously
**Handling**:
- PostgreSQL sequences are atomic — no custom locking needed
- Each thread gets unique value via `nextval()`
- No race conditions

## Appendix: Format Rationale

### Why Include Timestamp?
- Provides human-readable creation time without year/month rollover complexity
- Sortable (higher timestamp = later creation)
- Useful for debugging ("when was PR-ACME-1712345678901-00042 created?")

### Why Include Tenant ID?
- Multi-tenant isolation (ACME's PRs don't collide with TECH's)
- Human-readable (operators know which tenant without querying)
- Matches ADR-001 principle (tenant isolation at all layers)

### Why Global Sequence (no reset)?
- Simpler than annual/monthly reset logic
- Prevents gap issues ("year changed, sequences should reset, but which year?")
- Monotonic (numbers only increase, easier debugging)
- Tested pattern (used in SAP, Oracle, most enterprise ERPs)

### Why Lazy Sequence Creation?
- No upfront tenant provisioning required
- Works for any tenant ID immediately
- Minimal database overhead (sequences are cheap)
